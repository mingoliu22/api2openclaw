# Obsidian Writer

基于 Andy Matuschak 的 Evergreen Notes 和 Niklas Luhmann 的 Zettelkasten 方法论，帮助在 Obsidian 中构建知识网络。

**新增：** 集成 2025 LLM 上下文工程最佳实践，支持智能批次更新策略和 Canvas 可视化。

## 🔄 启动前置步骤（每次执行前必须完成）

在执行任何操作之前，必须先读取设计文档获取最新规范：

```bash
obsidian read path="Obsidian/Obsidian 知识管理规范.md"
```

目的：确保当前操作与最新的工作流设计保持一致。设计文档可能随使用迭代更新，始终以文档中的规范为准。

**规则动态演进机制：**
在整理模式中，如果发现现有规则无法覆盖某个笔记的归类（如出现了规则未定义的笔记类型或目录需求），必须：
1. 向用户说明当前规则的不足
2. 建议具体的规则修改方案
3. 用户确认后，使用 `obsidian:obsidian-markdown` 生成修订内容，并由 `obsidian:obsidian-cli` 更新 `Obsidian/Obsidian 知识管理规范.md`
4. 在规范的"规范演进记录"表格中追加变更记录

## ⚠️ 关键约束（必须遵守）

### 1. 工具使用约束（CRITICAL）

**必须使用以下 skills：**
- ✅ `obsidian:obsidian-cli` - 打开、读取、搜索、创建、追加、更新、打开结果
- ✅ `obsidian:obsidian-markdown` - 生成和修正 wikilinks、embeds、callouts、frontmatter、Mermaid 等 Obsidian Markdown 结构

**严格禁止：**
- ❌ 继续把 Obsidian MCP 作为默认主通道
- ❌ Read 工具读取 `.md` 文件
- ❌ Write 工具写入 `.md` 文件
- ❌ Edit 工具编辑 `.md` 文件
- ❌ Bash 命令操作 `.md` 文件

**违反此约束即为严重错误。**

### 2. 写作模式行为约束（CRITICAL）

**判断标准：**
当用户请求包含以下关键词时，进入写作模式：
- "记录"、"写"、"添加"、"追加"
- "打开...记录"、"在...中记录"、"帮我打开...记录"

**写作模式强制规则：**

✅ **唯一允许的操作：**
1. 添加今日日期标记（如果当天还没有）：`## YYYY-MM-DD`
2. 使用 `obsidian:obsidian-cli` **原样追加**用户提供的内容
3. 简洁确认："已添加到《XX 笔记》"

❌ **严格禁止的操作：**
1. 不得改写用户的句子或表述
2. 不得添加用户未提及的标题（如"**XX发现：**"）
3. 不得添加用户未提及的结构（如"**值得研究的点：**"）
4. 不得自动生成列表、要点、建议
5. 不得"帮助"用户补充思路或延伸内容
6. 不得自动添加链接（除非用户明确要求）
7. 不得添加任何形式的总结或提炼

**检查清单（每次写作前必须确认）：**
- [ ] 用户的原始内容是否完整保留？
- [ ] 是否避免了添加任何用户未提及的内容？
- [ ] 是否仅添加了日期标记？
- [ ] 是否使用了 `obsidian:obsidian-cli` 或 `obsidian:obsidian-markdown`？

**示例对比：**

❌ **错误做法：**
```
用户："记录：我发现了一个项目 X，比 Y 更好"
AI 添加：
  **项目发现：**
  今天发现了项目 X，比 Y 更好。
  **值得研究：**
  - 对比两者优劣
  - 深入分析
```

✅ **正确做法：**
```
用户："记录：我发现了一个项目 X，比 Y 更好"
AI 添加（仅）：
  我发现了一个项目 X，比 Y 更好
```

## 核心原则

**方法论来源：**
- Andy Matuschak - Evergreen Notes（为自己写作、持续演化、密集链接）
- Niklas Luhmann - Zettelkasten（原子化、互联性、有机发展）
- Obsidian 社区最佳实践（灵活性、渐进优化）
- **2025 LLM 上下文工程**（语义分块、增量更新、上下文预算管理）

**四大核心：**
1. **原子化**：一个笔记一个概念，自成体系
2. **链接网络**：通过 `[[双向链接]]` 建立关联，而非文件夹分类
3. **渐进演化**：从简单开始，随使用逐步精炼
4. **智能批次**：大量更新时自动按语义分批，避免上下文溢出

## 工作流程

### 1. 捕获阶段（写作模式）

**触发关键词：**
- "记录"、"写"、"添加"、"追加"
- "打开...记录"、"在...中记录"

**执行步骤：**

**场景 A：更新锚点笔记**
```
1. 使用 `obsidian:obsidian-cli` 读取 打开目标笔记
2. 检查是否有今日标记（## YYYY-MM-DD），无则添加
3. 使用 `obsidian:obsidian-cli` 追加 原样追加用户内容
4. 简洁确认："已添加到《XX 笔记》"
```

**场景 B：快速捕获临时想法**
```
1. 在 收件箱/ 创建笔记
2. 文件名：YYYYMMDD-HHMM-简短描述.md
3. 使用 `obsidian:obsidian-cli` 创建 创建
4. 告知用户位置
```

**重要提醒：写作模式下，严格遵守上述"写作模式行为约束"。**

---

### 2. 提炼阶段（整理模式）⭐ 新增智能批次更新

**触发关键词：**
- "整理"、"提炼"、"优化"、"重构"、"入库"

**只有用户明确使用这些关键词时才进入整理模式。**

#### 🆕 阶段 2.1：智能批次规划（预检查）

**执行步骤：**
```
1. 分析任务规模，识别需要操作的笔记数量
   - 需要新建的概念笔记
   - 需要更新的现有笔记
   - 需要更新的汇总/索引笔记

2. 判断是否需要批次处理：
   ✅ 触发批次模式的条件（满足任一）：
   - 操作数量 > 3 个笔记
   - 用户明确提到"批次"、"分批"、"避免上下文溢出"
   - 涉及大量内容整理（如"入库"、"整理大量资源"）

   ❌ 不需要批次的情况：
   - 仅 1-2 个笔记的简单操作
   - 用户明确要求"一次性完成"

3. 如触发批次模式，向用户展示规划：
   "📊 内容分析完成，建议分 N 个批次更新：

   批次 1: 新建概念笔记（X 个）
   批次 2: 更新锚点笔记（Y 个）
   批次 3: 更新汇总笔记（Z 个）

   这样可以避免上下文溢出。现在开始执行？"

4. 获得用户确认后，进入批次执行阶段
```

**批次划分原则（基于语义分块策略）：**
- **批次 1**：新建独立概念笔记（原子笔记）
- **批次 2**：更新现有主题笔记（锚点笔记）
- **批次 3**：更新汇总/索引/知识图谱笔记

每个批次独立完成后，向用户报告进度。

#### 阶段 2.2：执行批次更新（增量更新策略）

**批次 1: 新建概念笔记**
```
对每个待创建的概念：
1. 在 概念库/ 创建：YYYYMMDD-HHMM-概念名称.md
2. 使用原子笔记模板（见下方）
3. 使用 `obsidian:obsidian-cli` 创建 创建
4. 完成后报告："✅ 批次 1/N 完成 - 已创建 X 个概念笔记"
```

**批次 2: 更新锚点笔记**
```
对每个待更新的锚点笔记：
1. 使用 `obsidian:obsidian-cli` 读取 读取现有内容
2. 使用 `obsidian:obsidian-cli` 追加 追加新内容
   （或用 patch 替换特定部分）
3. 在新增内容中添加 [[概念链接]]
4. 完成后报告："✅ 批次 2/N 完成 - 已更新 Y 个锚点笔记"
```

**批次 3: 更新汇总笔记**
```
对知识图谱/索引笔记：
1. 使用 `obsidian:obsidian-cli` 追加 在末尾追加补充章节
   （标题如：## 📋 最新补充（YYYY-MM-DD））
2. 避免全量重写，仅追加新增部分
3. 完成后报告："✅ 批次 3/N 完成 - 已更新汇总笔记"
```

**重要原则（基于增量更新策略）：**
- ✅ 优先使用 `obsidian:obsidian-cli` 追加内容
- ✅ 必要时使用 `obsidian:obsidian-cli` 做局部更新
- ❌ 避免全量读取后整体重写（防止上下文溢出）
- ✅ 每批次完成后向用户报告进度

#### 阶段 2.3：传统提炼流程（单笔记优化）

**当不触发批次模式时，使用传统流程：**
```
1. 使用 `obsidian:obsidian-cli` 读取 读取锚点笔记
2. 分析内容，识别可独立的概念（满足原子化标准）
3. 向用户展示建议清单：
   "发现以下内容可以独立成概念笔记：
    1. 'XX概念' - 已有约 300 字描述
    2. 'YY方法' - 包含 3 个具体示例

    要帮你提取吗？"
4. 获得确认后，对每个概念：
   a. 在 概念库/ 创建：YYYYMMDD-HHMM-概念名称.md
   b. 使用原子笔记模板（见下方）
   c. 在原锚点笔记中替换为 [[概念名称]]
   d. 在概念笔记中添加：来源：[[锚点笔记名称]]
5. 确认："已提取 N 个概念到概念库"
```

**原子笔记模板：**
```markdown
# 概念名称

#标签1 #标签2

## 定义

[用自己的话解释这个概念]

## 核心要点

- 要点 1
- 要点 2

## 相关概念

- [[相关概念1]]
- [[相关概念2]]

## 来源

- [[锚点笔记名称]] - YYYY-MM-DD
```

**质量标准：**
- ✅ 聚焦单一概念
- ✅ 未来可独立理解（自成体系）
- ✅ 用自己的话表达（非复制粘贴）
- ❌ 避免：多个概念混杂
- ❌ 避免：仅仅是信息摘录

---

### 3. 连接阶段

**触发时机：**
仅在提炼阶段或用户明确要求时建立链接。

**执行步骤：**

```
1. 使用 `obsidian:obsidian-cli` 搜索 搜索相关概念
2. 发现潜在关联时建议：
   "这个概念与 [[已有概念]] 相关，要建立链接吗？"
3. 获得确认后，使用 `obsidian:obsidian-cli` 更新 添加链接
4. 在两个笔记中都添加双向链接
```

**链接原则：**
- ✅ 链接有意义（解释为什么相关）
- ✅ 跨主题连接（发现意外关联）
- ❌ 避免：为了链接而链接
- ❌ 避免：孤儿笔记（无任何链接）

---

### 4. 检索和聚合阶段

**场景 A：简单打开**
```
用户："打开《Claude 学习笔记》"
执行：使用 `obsidian:obsidian-cli` 读取 打开并展示
```

**场景 B：语义搜索**
```
用户："找找我之前记的关于提示词的内容"

执行步骤：
1. 使用 `obsidian:obsidian-cli` 搜索 进行语义搜索
2. 返回结果列表
3. 可选：建议建立关联
```

**场景 C：生成 Mermaid 思维导图**
```
用户："把《Claude 学习笔记》整理成思维导图"

执行步骤：
1. 使用 `obsidian:obsidian-cli` 读取 读取锚点笔记
2. 追踪所有 [[链接]]，递归读取链接的笔记
3. 分析内容结构和关系
4. 生成 Mermaid 思维导图代码
```

**场景 D：创建 Canvas 可视化** ⭐ 新增

```
触发关键词：
- "创建 Canvas"、"可视化"、"画布"
- "项目概览 Canvas"、"知识网络图"、"看板"

执行步骤：
1. 需求确认阶段（MUST DO）
   使用 AskUserQuestion 确认：
   - 要展示什么内容？（笔记网络/项目列表/概念地图/看板）
   - 数据来源？（特定笔记/搜索结果/手动指定）
   - 布局偏好？（网络图/分组/时间线/看板）

2. 数据收集阶段（MUST DO）
   a. 使用 `obsidian:obsidian-cli` 搜索
   b. 逐个使用 `obsidian:obsidian-cli` 读取相关笔记的完整内容
   c. 提取关键信息：
      - 标题
      - 完整描述（从原文提取，不缩写）
      - 链接、标签、元数据
   d. 列出所有将要包含的项目，询问用户确认是否完整

3. 调用 `obsidian:`obsidian:json-canvas`` 技能（CRITICAL）
   ⚠️ 重要：不要在 obsidian-writer 中重复实现 Canvas 创建逻辑

   使用 Skill 工具调用：
   Skill(skill="obsidian:`obsidian:json-canvas`", args="[准备好的数据和布局需求]")

   `obsidian:`obsidian:json-canvas`` 技能会：
   - 根据数据和布局要求生成 JSON Canvas 文件
   - 处理节点布局、尺寸、间距
   - 创建边线连接
   - 验证 Canvas 规范

4. 打开并确认
   - 使用 `obsidian:obsidian-cli` 打开创建的 Canvas
   - 询问用户反馈
   - 如需调整，再次调用 `obsidian:`obsidian:json-canvas`` 技能
```

**Canvas 创建质量标准（传递给 `obsidian:`obsidian:json-canvas`` 技能）：**
- ✅ 所有相关内容都已包含（无遗漏）
- ✅ 节点文字完整显示（不截断）
- ✅ 节点间距合理（不重叠）
- ✅ 分组清晰（不混乱）
- ✅ 描述完整（从原文提取，不缩写）

**Canvas 常见用途：**
- 项目看板（分组：待办/进行中/完成）
- 知识网络图（展示概念间的链接关系）
- 研究画布（中心问题 + 相关资料 + 参考文献）
- 学习路线图（按时间线或主题组织）

## 笔记组织结构

```
/
├── 主笔记/              # 锚点笔记（持续更新的学习日志）
├── 概念库/              # 原子笔记（提炼的独立概念）
├── 收件箱/              # 临时捕获（未分类内容）
├── Obsidian/            # 工具文档（配置指南、使用规范）
├── Claude/              # Claude 相关（使用经验、SOP、项目方案）
├── 资源/               # 附件和模板
└── 知识库索引.md         # 根目录唯一例外（全局导航入口）
```

**命名规范：**
- 锚点笔记：`主题名称.md`
- 原子笔记：`YYYYMMDD-HHMM-概念名称.md`
- 临时笔记：`YYYYMMDD-HHMM-描述.md`
- Canvas 文件：`主题名称-Canvas.canvas` 或 `YYYYMMDD-描述.canvas`

## 技能协作

### 与 `obsidian:json-canvas` 技能的协作

**职责分工：**
- **obsidian-writer**：负责数据收集、需求分析、用户确认
- **`obsidian:json-canvas`**：负责 Canvas 文件的创建、布局设计、规范验证

**协作流程：**
```
1. obsidian-writer 收集数据
   ├─ 使用 `obsidian:obsidian-cli` 搜索和读取笔记
   ├─ 提取完整信息
   └─ 整理为结构化数据

2. obsidian-writer 调用 `obsidian:json-canvas` 技能
   ├─ 传递数据和布局需求
   └─ `obsidian:json-canvas` 生成 Canvas JSON

3. obsidian-writer 使用 `obsidian:obsidian-cli` 创建文件

4. obsidian-writer 打开并收集反馈
   └─ 使用 `obsidian:obsidian-cli` 打开结果
```

**重要原则：**
- ❌ 不要在 obsidian-writer 中重复实现 Canvas 创建逻辑
- ✅ 充分利用 `obsidian:json-canvas` 技能的专业能力
- ✅ obsidian-writer 专注于知识管理工作流
- ✅ `obsidian:json-canvas` 专注于 Canvas 规范和布局

## 常见错误避免

**❌ 不要做：**
- 在写作模式下添加用户未提及的内容
- 在写作模式下改写用户的表述
- 在写作模式下"帮助"用户完善思路
- 使用 Read/Write/Edit 工具操作 .md 文件
- 混淆写作模式和整理模式
- **大量更新时一次性读写全部内容（会导致上下文溢出）**
- **在 obsidian-writer 中重复实现 Canvas 创建逻辑**

**✅ 要做：**
- 写作时忠实记录用户原文
- 整理时主动建议提炼和优化
- 始终优先使用 `obsidian:obsidian-cli` 与 `obsidian:obsidian-markdown`
- 明确区分不同工作模式
- **大量更新时自动触发批次模式，分批执行**
- **优先使用 append 和 patch，避免全量重写**
- **创建 Canvas 时调用 `obsidian:json-canvas` 技能**

## 智能批次更新检查清单

执行整理模式前，必须确认：
- [ ] 是否分析了操作规模（需要操作多少个笔记）？
- [ ] 如果 > 3 个操作，是否向用户展示了批次规划？
- [ ] 是否按语义边界合理划分批次？
- [ ] 每个批次是否使用了增量更新（append/patch）？
- [ ] 每批次完成后是否向用户报告进度？
- [ ] 是否避免了全量读取和重写？

---

## Canvas 创建检查清单

创建 Canvas 前，必须确认：
- [ ] 是否使用 AskUserQuestion 确认用户真实需求？
- [ ] 是否完整搜索所有相关笔记（不遗漏）？
- [ ] 是否逐个读取笔记完整内容（不只读标题）？
- [ ] 是否列出将要包含的所有项目，询问用户确认？
- [ ] 是否从原文提取完整描述（不缩写）？
- [ ] 是否调用了 `obsidian:json-canvas` 技能（而非自己实现）？
- [ ] 是否使用 `obsidian:obsidian-cli` 打开 Canvas 并收集反馈？
